## Goal

Build a production-ready Smart Contract Security Analyzer as an MCP server on Vercel using Next.js App Router. The app exposes 4 MCP tools (analyze_contract, check_vulnerability, generate_audit_report, suggest_fix) backed by 7 AST-based Solidity vulnerability detectors, ships a marketing landing page, an interactive web demo playground, and a direct REST analysis endpoint for the web demo. Everything must build cleanly, validate with zod, and demonstrably detect the four required vulnerability classes in the canonical VulnerableBank example.

## Context

- **Workspace root:** `/home/harness/cyops_data/workspace/smart-contract-mcp`. The project is greenfield — no prior code exists at this location other than this planning artifact.
- **MCP transport:** Vercel serverless functions do not support stdio transports, only Streamable HTTP. We must use Vercel's `mcp-handler` package, which wraps `@modelcontextprotocol/sdk` with a Next.js App Router route handler that supports `GET`, `POST`, and `DELETE` methods.
- **Analysis runtime:** Vercel does not allow Python subprocesses (e.g. `subprocess.run(['slither', ...])`). All detection must be pure TypeScript, using `@solidity-parser/parser` to produce an AST and visitor-walk it.
- **Pragma semantics:** Solidity ≥ 0.8.0 reverts on integer overflow by default, so the integer-overflow detector must skip those contracts unless an `unchecked {}` block is present.
- **Component library:** The task lists shadcn/ui-style primitives (Button, Card, Textarea, Badge, Tabs) but the dependency list only ships `class-variance-authority`, `clsx`, and `lucide-react`. We will hand-author small, styled-by-convention components in `components/` rather than running the shadcn CLI (which requires interactive setup and is not available in non-TTY contexts).
- **Acceptance bar:** The seven ACs in the original brief are the test plan. The VulnerableBank contract MUST trigger ≥ 4 findings (reentrancy, tx.origin, integer overflow, unprotected selfdestruct) with the right severities so AC-3, AC-4, and the implied severity-coverage claims all pass.

## Assumptions

- The shadcn/ui components are authored as plain Tailwind + cva React components in `components/`, not generated via `npx shadcn@latest add`. This matches the dependency list provided.
- The `mcp-handler` package's documented API (`createMcpHandler(serverFn, options, init)` returning a Next.js route handler usable as `GET`/`POST`/`DELETE`) is used as the canonical signature. If the installed version differs, the implementer pins the version that matches the API used here.
- `next.config.ts` is written as a TypeScript Next config (Next 14.2+ supports this). If the build environment rejects the `.ts` extension, fall back to `next.config.mjs` with the same options.
- Line numbers for findings come from the parser's `loc` field, enabled by `parser.parse(source, { loc: true, tolerant: true })`. Locations on the function call / member access / identifier nodes are used, not on the whole expression.
- Pragma version is detected by a single regex on the raw source: `/pragma\s+solidity\s+(\^|>=|<=|=)?\s*(\d+)\.(\d+)\.(\d+)/`. We compare the major+minor tuple. A pragma of `^0.6.0` is treated as < 0.8.0 (vulnerable). Absence of a pragma is treated conservatively as < 0.8.0 (flag arithmetic) so that unannotated snippets still get audited.
- The `mcp-handler` basePath is `/api` so the full MCP URL is `/api/mcp`.
- Vercel deployment uses the default build command; no custom install scripts are required.
- No external network calls at runtime — all detection is local AST analysis.

## Implementation Notes

### Build order (top-down dependency graph)

1. **Configuration files first** — they shape the rest of the work.
   - `package.json` with the exact dependency list from the brief. Use `next@14.2.18` (the last 14.2.x stable that supports `next.config.ts` cleanly) and pin `mcp-handler` to `^1.0.0` once the API is verified locally; if the package's `latest` is already ≥ 1.0, use `latest`.
   - `tsconfig.json` extending `next/core-web-vitals`, `"strict": true`, `"moduleResolution": "bundler"`, `"target": "ES2022"`.
   - `tailwind.config.ts` with content globs `./app/**/*.{ts,tsx}` and `./components/**/*.{ts,tsx}`, dark-mode class strategy, and the custom purple accent color `accent: { DEFAULT: "#8B5CF6" }` in the theme extension.
   - `next.config.ts` exporting a default config with `experimental.serverComponentsExternalPackages: ["@solidity-parser/parser"]` so the parser is not bundled into the client.
   - `vercel.json` exactly as specified in the brief.
   - `postcss.config.mjs` with `tailwindcss` and `autoprefixer` plugins.
   - `app/globals.css` with `@tailwind base; @tailwind components; @tailwind utilities;`, a body background of `bg-gray-950`, and CSS variables for the accent color.

2. **Shared types** (`lib/types.ts`) — exported verbatim from the brief: `Severity`, `Vulnerability`, `AnalysisResult`. Add a `DetectedFinding` internal type used by detectors before they assemble the public `Vulnerability` shape, so detectors can carry a raw `node` for the suggest_fix pass.

3. **Parser wrapper** (`lib/analyzers/parser.ts`) — exports `parseSolidity(source: string): { ast: ASTNode; source: string; pragma?: { major: number; minor: number; patch: number } }`. Wraps `parser.parse(source, { loc: true, tolerant: true })` and runs the pragma regex. Throws a typed `ParseError` on failure that the API route catches and surfaces as a 400.

4. **Detectors** (one file each, plus `index.ts`). Each detector exports `detect(ctx: DetectorContext): Vulnerability[]`. The `DetectorContext` carries the AST, the raw source, the parsed pragma, and a `functionStack: string[]` (so detectors can attribute findings to the enclosing function name).
   - `reentrancy.ts`: Walk `FunctionDefinition` nodes. For each function, scan the `body` statements in order. Record the first statement that contains a `FunctionCall` whose `expression` ends in `.call{...}(...)`, `.transfer(`, `.send(`, or low-level `call(`. Then look for any subsequent `BinaryOperation` or `Assignment` that writes to a `MemberAccess` on `this` (state write). If found, emit a `critical` `SWC-107` finding with the function name and the lines of both the call and the late write.
   - `access-control.ts`: Walk `FunctionDefinition` with `visibility ∈ {public, external}` and `stateMutability !== "view" !== "pure"`. For each, check whether the body contains any expression that reads `msg.sender` (a `MemberAccess` of `msg.sender` inside a `FunctionCall` argument or a `BinaryOperation`). If not, and the function writes to state, emit a `high` `SWC-105` finding. Skip functions whose name is `constructor`. Skip pure event-emitting functions by checking that the body references a state variable assignment.
   - `integer-overflow.ts`: First, branch on pragma. If `pragma.major < 0.8` or pragma is missing, walk all `BinaryOperation` nodes with operator in `+ - *` where both operands are `uint`/`int` typed (best-effort: use the typeString of any `VariableDeclaration` they touch; otherwise assume `uint256` and flag — false positives are acceptable here). If `pragma.major >= 0.8`, only flag arithmetic inside an `UncheckedStatement` (`UncheckedBlock` in the AST) — collect the inner `BinaryOperation`s and emit a finding per function.
   - `low-level-calls.ts`: Walk `FunctionCall` nodes whose `expression` matches `MemberAccess` with memberName in `{call, delegatecall, staticcall}`. Look at the parent statement: if it is a `VariableDeclarationStatement` like `(bool success, ) = ...` and the next statement in the enclosing block is a `FunctionCall` to `require(success)` OR the function call itself is wrapped in `require(...)`, mark it as checked. Otherwise, emit a `medium` `SWC-104` finding.
   - `timestamp.ts`: Walk `MemberAccess` nodes with `memberName === "timestamp"` and `expression` is the identifier `block` (or the legacy alias `now`). If the parent is a `BinaryOperation` with operator in `== != < > <= >=`, emit a `low` `SWC-116` finding. Also flag usages as a randomness seed (any `BinaryOperation` with `%` or `keccak256(...)` argument that includes `block.timestamp`).
   - `tx-origin.ts`: Walk `MemberAccess` nodes with `memberName === "origin"` and `expression` is the identifier `tx`. If the nearest enclosing function is being used for authorization (the `MemberAccess` appears as an argument to a `FunctionCall` whose callee name is `require`, or as the test of an `IfStatement`), emit a `high` `SWC-115` finding.
   - `selfdestruct.ts`: Walk `FunctionCall` nodes whose callee is the identifier `selfdestruct` (Solidity ≥ 0.8 allows `selfdestruct(...)`; pre-0.8 also allows `suicide(...)` — flag both). For each, walk up to the enclosing `FunctionDefinition` and apply the same access-control check as the access-control detector. If unprotected, emit a `critical` `SWC-106` finding.

5. **Aggregator** (`lib/analyzers/index.ts`):
   - Exports `analyze(source: string, contractName?: string): AnalysisResult`.
   - Calls `parseSolidity`, then runs all seven detectors.
   - Derives `contractName`: prefer the first `ContractDefinition` name in the AST; if `contractName` arg is provided, use that.
   - Builds `summary` from a templated string that names the contract and the count of critical/high findings ("Contract `X` has N critical and M high-severity issues. Immediate remediation is recommended.").
   - Sets `analyzedAt` to `new Date().toISOString()`.
   - Returns the full object.

6. **Risk scoring** (`lib/scoring.ts`):
   - Exports `calculateRiskScore(vulnerabilities: Vulnerability[]): number` and `riskLevel(score: number): "low" | "medium" | "high" | "critical"`.
   - Iterate severities, accumulate by the cap table in the brief, then clamp to 100.
   - `riskLevel` uses the four bands exactly as specified.

7. **Direct REST route** (`app/api/analyze/route.ts`):
   - `POST` handler that reads `{ source_code, contract_name? }` from a zod-validated JSON body.
   - Calls `analyze()` from `lib/analyzers/index.ts` and returns the `AnalysisResult` as JSON.
   - Wraps `parseSolidity` errors in a 400 response with `{ error: "parse_error", message }`.

8. **MCP route** (`app/api/mcp/route.ts`):
   - Use `createMcpHandler((server) => { ... }, { ... }, { basePath: "/api" })` and re-export as `GET`, `POST`, `DELETE`.
   - Tool 1 `analyze_contract`: schema `{ source_code: z.string(), contract_name: z.string().optional() }`, runs the same `analyze()` and returns `{ content: [{ type: "text", text: JSON.stringify(result) }] }`.
   - Tool 2 `check_vulnerability`: schema `{ source_code: z.string(), vulnerability_type: z.enum(["reentrancy","access_control","integer_overflow","unchecked_calls","timestamp_dependence","tx_origin","selfdestruct"]) }`, runs `analyze()`, filters `vulnerabilities` to those whose `id` matches the type, and returns them.
   - Tool 3 `generate_audit_report`: schema `{ source_code: z.string(), contract_name: z.string(), auditor_name: z.string().optional() }`. Builds a markdown string with sections (Executive Summary, Risk Score, Findings Table, Detailed Findings, Recommendations). The detailed findings section includes the vulnerable code slice (3 lines around each finding's first line) and the remediation text.
   - Tool 4 `suggest_fix`: schema `{ source_code: z.string(), vulnerability_id: z.string(), vulnerability_name: z.string() }`. Re-runs `analyze()` to find the matching finding, extracts the affected source slice (the enclosing function body), and returns a hand-written `fixed_code` for the canonical pattern: reentrancy → Checks-Effects-Interactions; tx.origin → msg.sender; integer overflow → Solidity 0.8 pragma or SafeMath; selfdestruct → `onlyOwner` modifier; timestamp → block.number; unchecked call → `require(success, ...)`. The `diff_summary` lists the changes in plain English.

9. **UI components** (in `components/`):
   - `ContractInput.tsx` — controlled `<textarea>` plus an "Analyze" button. Accepts `defaultValue`, `onAnalyze(source: string)`, and `isLoading` props. Uses Tailwind classes for the dark theme.
   - `VulnerabilityCard.tsx` — renders one `Vulnerability` with severity-colored `Badge`, SWC link, location, description, and remediation.
   - `RiskScore.tsx` — circular gauge (an SVG `<circle>` with `strokeDasharray`) whose color band matches `riskLevel(score)`.
   - `AuditReport.tsx` — renders an `AnalysisResult` by composing `RiskScore` + a list of `VulnerabilityCard`s + a "Copy as JSON" button.
   - `InstallInstructions.tsx` — renders the JSON config block in a `<pre>` with a copy-to-clipboard button.

10. **Pages**:
    - `app/layout.tsx` — root layout with `<html className="dark">`, the body class for gray-950, and a global header linking to `/` and `/demo`.
    - `app/page.tsx` — landing page (hero, three feature cards, install section, link to `/demo`).
    - `app/demo/page.tsx` — server component that preloads the VulnerableBank source into the page (passed as a prop to `ContractInput`). Client component renders the two-pane layout.

11. **Smoke-test scripts** (in `scripts/`):

    - `smoke-mcp.mjs` — issues a `POST /api/mcp` with the `analyze_contract` tool call and asserts the VulnerableBank findings.

### Key technical decisions

- **MCP tool return shape:** Always return `{ content: [{ type: "text", text: <string> }] }`. Stringify JSON inside `text`. This matches `mcp-handler` and `@modelcontextprotocol/sdk` expectations.
- **No new dependencies beyond the brief.** Detector logic must be hand-rolled against the AST visitor. Resist the urge to add `solc` or `slither` packages.
- **False-positive tolerance:** The brief explicitly accepts false positives — `integer-overflow` is allowed to flag any arithmetic in a `< 0.8` contract. We do not over-engineer type inference.
- **Tolerance for parser failures:** `tolerant: true` is set on `parser.parse` so most real-world snippets parse. The catch block in `parseSolidity` produces a clean error message for the API.
- **Dark theme + purple accent:** Implemented via Tailwind config theme extension and CSS variables in `globals.css`. No runtime theme switcher.

### Risk controls

- **MCP schema drift:** Pin `mcp-handler` after a successful local build; do not let CI resolve `latest` blindly. If `createMcpHandler`'s signature changes in a future major, the implementer should keep the API as documented here and pin the resolved version in `package.json` before commit.
- **TypeScript strictness:** Use `import type` for all type-only imports; explicitly type the `loc` field where used to avoid `any` leaks that would fail `tsc --strict`.
- **Vercel build size:** `@solidity-parser/parser` is marked as an external server-component package in `next.config.ts` to keep the client bundle clean.
- **VulnerableBank test reproducibility:** `scripts/vulnerable-bank-payload.json` is committed to the repo so AC-3 and AC-4 verifications are deterministic.

## Out of Scope

- Multi-file Solidity projects (imports, dependencies between files). The parser only sees one source string per request.
- Compiler-grade type inference. We use a "best effort + pragma-aware" model — false positives on overflow detection are accepted.
- Gas estimation, gas-optimization suggestions, or formal verification.
- On-chain lookups (proxy implementations, verified-source downloads from Etherscan). Detection is purely static.
- Persistent storage of analysis results, user accounts, or rate limiting beyond Vercel's defaults.
- Automated deployment from this session — the implementer authors `vercel.json` and the README, but the actual `vercel deploy` push is left to the user.
- Translating Solidity to other languages or generating upgradeable-contract patterns.
- Python tooling, slither/mythril integration, or any subprocess-based analyzer.
